# Cloudflare Page Rule Migration Advisor (AI-Powered)

**Live tool:** https://cf-pagerule-migrator.burningbros.workers.dev

A free tool that answers: *"I have this Page Rule — exactly which modern Cloudflare Rules do I need to recreate it as?"*

Paste your existing Page Rule's URL match pattern and its enabled settings (in your own words — typos and rephrasing are fine). You get back two independent things:

1. A **draft filter expression** for the modern Rules engine, generated by plain deterministic code (not AI) from your URL pattern, using Cloudflare's own `wildcard` operator syntax.
2. A **settings breakdown**, generated by an AI model grounded in Cloudflare's own official Page Rules migration table, that tells you exactly which Cache / Redirect / Configuration / Origin / Transform Rule each of your settings needs to move to (and flags anything fully deprecated with no replacement).

## Why this exists

Cloudflare stopped allowing new Page Rules on Free/Pro/Business plans in mid-2024 and on Enterprise in late 2024, and has been auto-migrating existing Page Rules to the modern Rules engine through 2025-2026. This generated a lot of confusion in the Cloudflare Community forums — threads titled "major confusion", "no good replacements?", "manual or auto?" are common. Cloudflare's own docs have a static, generic table (setting name -> rule category), but as of the last check (2026-07-13) there was no interactive tool that takes a user's *actual* Page Rule — their specific settings and URL pattern — and tells them exactly what they personally need to (re)build. (There is an unrelated tool also called "Rule Converter" at rule-converter-web.pages.dev, but it converts proxy/VPN routing rule formats like Clash/Mihomo — nothing to do with Cloudflare's CDN Rules engine.)

## How it works

- **Filter expression**: plain string logic splits your pasted URL pattern into host + path and builds a `(http.host eq/wildcard "...") and (http.request.uri.path wildcard "...")` expression using Cloudflare's documented `wildcard` operator (works on every plan, unlike the `matches` regex operator which needs Business+). No model call in this path — the syntax you see is deterministic, not a guess.
- **Settings breakdown**: a hardcoded table of all ~33 official Page Rule settings and their exact migration target, sourced from Cloudflare's own [Page Rules migration guide](https://developers.cloudflare.com/rules/reference/page-rules-migration/). [Cloudflare Workers AI](https://developers.cloudflare.com/workers-ai/) (`@cf/meta/llama-3.1-8b-instruct-fp8-fast`) matches your free-text settings against this table and reports only what's listed — it's explicitly instructed not to invent a setting or target, and given explicit disambiguation guidance for the four similarly-named HTTPS/TLS settings (Always Use HTTPS / Automatic HTTPS Rewrites / SSL / Opportunistic Encryption) that it initially confused with each other during testing.
- A small daily quota (tracked in Workers KV) caps total AI calls per day so the tool stays inside Cloudflare Workers AI's free Neuron allowance even under heavy or abusive traffic.

## Known limitation

This uses a small (8B) instruction-tuned model for the settings breakdown, not a large reasoning model. During testing it occasionally mismatched adjacent, similarly-named table rows (e.g. reporting "Always Use HTTPS" under the "SSL" row) before the system prompt was tightened with explicit per-row disambiguation. The filter expression (the part most likely to cause real harm if wrong) is deterministic code, not AI, specifically to avoid this risk. Always verify the AI-suggested rule groupings against the [official migration table](https://developers.cloudflare.com/rules/reference/page-rules-migration/) and the filter expression in the Cloudflare dashboard's rule builder before deploying anything to a live zone.

## Stack

- Single [Cloudflare Worker](https://developers.cloudflare.com/workers/) (`worker.js`), no framework, no build step.
- Bindings: Workers AI (`env.AI`) + one Workers KV namespace for the daily quota counter + one shared Workers KV namespace for the traffic counter.
- Deploy with [Wrangler](https://developers.cloudflare.com/workers/wrangler/):

```bash
npx wrangler kv namespace create PAGERULE_QUOTA   # then put the returned id in wrangler.toml
npx wrangler deploy
```

## Keeping the facts current

Cloudflare occasionally changes the migration table or the Rules language syntax. To refresh:

1. Check https://developers.cloudflare.com/rules/reference/page-rules-migration/ for the settings table, and https://developers.cloudflare.com/ruleset-engine/rules-language/ for filter expression syntax.
2. Update the `PAGERULE_MAP` array and/or `pageRuleToFilterExpr()` in `worker.js` (keep entries sourced only from official docs).
3. Redeploy.

PRs that add more official-doc-sourced facts or fix bugs are welcome.

## Traffic

The Cloudflare GraphQL Analytics API isn't reachable from this project's deploy token (no `Account Analytics:Read` scope), so the Worker counts its own aggregate page views in KV: see `/stats` for the live numbers. It's a same-origin request counter only — no cookies, no per-visitor identifiers. Requests sending an `X-Skip-Analytics: 1` header or a common bot/test User-Agent (curl, Playwright, Googlebot, Discordbot, etc.) aren't counted.
